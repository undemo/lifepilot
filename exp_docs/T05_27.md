# T05_27 活动/娱乐场所理解与召回修复

## 背景

上一轮 LLM-native 增量升级已经把食物表达编译成 `FoodIntent`，并通过 `MachineIntent` 影响召回和排序。但活动表达仍主要依赖旧 `intent_tags` 和少量 `SPORTS` 特判，导致“打羽毛球 / 打电竞 / 景点逛逛”等显式活动需求容易被默认场景排序覆盖。

## 诊断

核心问题不是缺几个关键词，而是缺一层与 `FoodIntent` 对称的活动内部契约：

- 用户活动词没有统一结构化为 activity type / parent category / facility / intensity / companion suitability。
- `CandidateRetriever` 把 sports、esports、script_murder、board_game 等默认视作 `low_fit_activity`，在 city/relationship 场景里会被强降权。
- POI feature store 已有很多语义标签，但缺 `activity_features` 这种可执行字段。
- 高德补候选如果直接按“运动/公园/商场”等宽词召回，会把不相关 POI 错标成目标活动。

## 解法

新增 `ActivityIntent`，把活动表达拆成：

- `raw_terms`
- `activity_type_ids`
- `parent_categories`
- `facility_types`
- `genres`
- `styles`
- `scenes`
- `intensity`
- `child_suitable_required`
- `elderly_suitable_required`
- `quiet_required`
- `social_mode`

检索仍然不让 LLM 扫全量 POI。流程是：

```text
IntentParser
-> ConstraintExtractor
-> LatentIntentInterpreter
-> FoodSemanticAgent
-> ActivitySemanticAgent
-> RetrievalIntentCompiler
-> CandidateRetriever(machine_intent)
-> VerifierService
```

活动和食物一样，只在 query side 做语义编译；事实来源仍是 MockAPI / POIFeatureStore / Verifier。

## 关键实现

- 新增 `backend/app/services/activity_semantic_agent.py`
- 新增 `backend/data/activity_semantics.json`
- 扩展 `backend/app/schemas/internal_intelligence.py`
  - `ActivityIntent`
  - controlled `ActivityTypeId`
  - controlled `ActivityParentCategory`
- 扩展 `backend/app/services/retrieval_intent_compiler.py`
  - 输出 `retrieval_plan.activity_match`
  - 输出 activity slot requirement
  - 加入儿童/长辈/安静/运动后补水等偏好和惩罚
- 扩展 `backend/app/services/candidate_retriever.py`
  - 只消费结构化 `MachineIntent`
  - 泛化 `_activity_match_score`
  - 显式活动需求时提升匹配活动，并压低不匹配活动
  - 对 child/elderly safety 做 deterministic demotion
- 扩展 `backend/app/rules/poi_feature_store.py`
  - 支持 `poi_activity_attributes.json`
  - 派生 `activity_features`
- 新增 `backend/data/poi_activity_attributes.json`
  - 覆盖羽毛球、电竞、剧本杀、电影、剧院、游乐园、手工、景点/步道等 demo POI
- 新增 `backend/app/services/gaode_activity_provider.py`
  - 本地 Gaode review candidate 优先
  - `AMAP_KEY` 存在时才调用高德文本搜索
  - 生成的候选写入 `runtime_activity_pois.json`
  - MockAPI 合并 runtime POI，保证后续状态/路线仍走 MockAPI

## 推演结果

- “和姐姐打羽毛球”
  - `ACTIVITY_BADMINTON / SPORTS / 羽毛球馆`
  - 选中羽毛球馆。
- “和室友去打电竞”
  - `ACTIVITY_ESPORTS / GAME / SOCIAL_ENTERTAINMENT`
  - 非亲子/长辈场景可选电竞。
- “和朋友去景点逛逛”
  - `ACTIVITY_SCENIC + ACTIVITY_PARK_WALK`
  - 优先步道/湖畔/公园，不再被普通商场或写真影棚抢走。
- “今天下午想喝奶茶”
  - ActivityIntent 保持 unknown
  - FoodIntent 命中奶茶/饮品，餐饮候选命中茶姬。
- “孩子5岁想打电竞”
  - 电竞被安全降权，亲子友好活动优先。
- “带爷爷去踢足球”
  - 高强度对抗运动被长辈安全规则降权。

## 试用前补丁

手工 smoke 时发现一个泛化误伤：`low_fit_activity` 曾被当作电竞/剧本杀的兜底标签，导致 KTV、健身、攀岩这类“低适配娱乐”在“打电竞”请求下被错误算作匹配候选。

修复：

- `ACTIVITY_ESPORTS` 只允许 `esports` 标签或“电竞/网咖/网吧”等明确文本命中。
- `ACTIVITY_SCRIPT_MURDER` 只允许剧本杀/桌游/推理等明确标签或文本命中。
- `GAME` 父类不再用 `low_fit_activity` 做宽匹配。
- 有明确活动意图时，非匹配活动不再进入首选；只有儿童/长辈安全冲突场景才允许安全替代。

验证：

```text
PYTHONPATH=backend backend/.venv/bin/python -m pytest -q \
  tests/test_candidate_retriever_activity_intent.py \
  tests/test_llm_native_activity_flow.py

10 passed
```

稳定 demo 时间 `LIFEPILOT_DEMO_NOW=2026-05-21T13:30:00+08:00` 下，“今晚和室友去打电竞”命中 `杰拉电竞·网咖·精选(杭州吾角天街店)`。如果真实当前时间导致所有电竞馆不可用，不再用写真/攀岩等不相关活动冒充首选。

## 风险控制

- 不改 `/api/v1`。
- 不改 PlanContract 既有必填字段。
- 新字段只在 internal / optional 路径使用。
- CandidateRetriever 不依赖 LLM。
- 高德补候选是 optional fallback，没有 `AMAP_KEY` 不影响 demo。
- LLM disabled 下全链路可跑。
- Verifier 仍是最终事实闸门。

## 后续建议

- 前端可选展示 `projection.reason_cards` 时，补一张“活动匹配理由”卡片。
- 后续可把 `ActivityIntent` 传给 `ExplanationAgent`，让中文解释能更明确说明“为什么适合这类同行人”。
- 高德补候选建议加人工 review 开关，避免宽查询词污染 runtime POI。
